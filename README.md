# BOLA / IDOR — навчальний стенд (OWASP API)

Повноцінний локальний проєкт: React-клієнт, API Gateway з перевіркою JWT, сервіс автентифікації, сервіс звітів про вразливості, PostgreSQL, Docker Compose. Демонструється **порушення авторизації на рівні об’єкта (BOLA)**: автентифікований користувач отримує **чужий** запис за прямим ID у URL, якщо бекенд не звіряє власника об’єкта з клеймами JWT.

## Архітектура

- **web** (nginx + React): SPA на `http://localhost:3000`, проксує `/api/*` на gateway.
- **gateway** (FastAPI): перевіряє підпис JWT (`org_id`, `username`), додає внутрішні заголовки `X-Gateway-Token` та `X-Trusted-Org-Id`, проксує до auth/reports. Клієнт **не** надсилає ці секрети.
- **auth**: `POST /auth/login` (OAuth2 password), `GET /auth/me` (внутрішньо з gateway за `username` з токена).
- **reports**: `GET /reports`, `GET /reports/{id}` — доступ лише з gateway; режим `REPORTS_AUTHZ_MODE` керує перевіркою `org_id` запису проти довіреного org з JWT.

## Швидкий старт

1. Скопіюйте `.env.example` у `.env` і задайте довгі випадкові `JWT_SECRET` та `GATEWAY_INTERNAL_TOKEN` (не використовуйте прикладові значення в публічних середовищах).

2. Запуск:

   ```bash
   docker compose up --build
   ```

3. Відкрийте `http://localhost:3005` (порт налаштовано в `docker-compose.yml`, за замовчуванням 3005 якщо 3000 зайнятий), увійдіть тестовим користувачем (див. нижче). На головній сторінці список містить лише звіти **вашої** організації. Натисніть «Відкрити /reports/[id]» з ID **201** (звіт організації Beta) — на сторінці деталей з’явиться попередження про BOLA, якщо увімкнено вразливий режим.

4. Прямий доступ до API gateway з хоста (для скриптів): `http://127.0.0.1:23456` (у контейнері gateway слухає 8080).

## Режими звітів

| Змінна | Поведінка |
|--------|-----------|
| `REPORTS_AUTHZ_MODE=vulnerable` | Повертає звіт за `id` без перевірки належності до `org_id` з JWT — **класичний BOLA/IDOR** при наявній автентифікації. |
| `REPORTS_AUTHZ_MODE=secure` | Порівнює `report.org_id` з довіреним org від gateway; чужий об’єкт → **403**. |

Перемикання без редагування `.env` вручну:

```bash
docker compose -f docker-compose.yml -f docker-compose.secure.yml up --build
```

## Тестові облікові записи

Паролі задані лише для локального навчання; у продакшені так не роблять.

| Користувач | Пароль | org_id |
|------------|--------|--------|
| `alice` | `K7m!pQ2$vL9#` | 1 (Alpha) |
| `bob` | `R4n@xY8wZ1%` | 2 (Beta) |

Приклади ID звітів: **101–102** — org 1; **201–202** — org 2.

## Автоматизація атаки (CLI)

Після `docker compose up`:

```bash
python scripts/bola_attack_demo.py
```

Скрипт логіниться як `alice` і запитує звіт `201`. Очікування: у `vulnerable` — відповідь 200 з даними чужої org; у `secure` — 403.

## Сценарій для захисту

1. Gateway **обов’язково** перевіряє JWT і передає лише **довірений** `org_id` внутрішньому API (заголовок `X-Trusted-Org-Id`), а не значення з клієнта.
2. Мікросервіс звітів у **secure**-режимі виконує **обов’язкову** перевірку: ресурс має належати тій самій організації, що й у токені (захист від помилок у майбутніх клієнтах і прямих викликів за ID).

Додатково в проді: окремі мережі для внутрішніх сервісів, ротація секретів, аудит, обмеження швидкості, мінімізація даних у відповідях.

## Розробка фронтенду без Docker

Потрібні запущені backend-сервіси та gateway на `23456` (мапінг з контейнерного 8080):

```bash
cd frontend && npm install && npm run dev
```

Vite проксує `/api` на `http://127.0.0.1:23456`.
